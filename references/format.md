# 파일 포맷 상세 (work-done 스킬)

## 일일 파일 (YYYY-MM-DD.md)

### Frontmatter 스키마

```yaml
---
date: 2026-05-17              # ISO 형식 (YYYY-MM-DD)
topics: [SBR, python-script]  # 주제 태그 배열. 짧고 검색 가능한 키워드
projects: [WAI-Design]        # 프로젝트명 배열. 작업 폴더 경로에서 추론
sessions: 2                   # 이 날 /work_done이 호출된 횟수
---
```

규칙:
- `date`는 무조건 ISO 형식
- `topics`·`projects`는 항상 배열 (`[]` 사용). 단일 값이어도 배열.
- 태그는 공백 대신 하이픈 (`python-script`, `SBR-process`)
- 중복 제거. 정렬은 추가된 순서를 유지해도 OK.

### 본문 구조

```markdown
# YYYY-MM-DD 업무 기록

## [HH:MM] <세션 제목 — 한 일의 핵심을 한 줄로>

### 한 일 (What)
- 동사로 시작하는 구체적 변경/작업
- 파일명·함수명·수치 등 식별자 포함
- 한 줄당 하나의 행동

### 왜 (Why)
- 작업의 동기. 사용자가 명시한 맥락 우선.
- 해결하려던 문제, 사용자의 요청, 외부 제약 등

### 의사결정·트레이드오프
- "A vs B 중 A 선택. 이유: ..." 형태
- 사용자와 의논해서 정한 선택은 반드시 포함
- 미루기로 한 것도 의사결정이므로 적기

### 다음 (Next)
- 후속 작업, 미완료 항목, 사용자가 별도로 챙기기로 한 것
- 없으면 이 섹션 생략 가능
```

여러 세션이 있을 때는 `## [HH:MM]` 섹션을 `---` 구분선으로 나눠 누적:

```markdown
## [09:30] 첫 세션 제목

### 한 일 (What)
...

---

## [14:15] 두 번째 세션 제목

### 한 일 (What)
...
```

### 풀 예시

```markdown
---
date: 2026-05-17
topics: [SBR, python-script, refactor]
projects: [WAI-Design]
sessions: 2
---

# 2026-05-17 업무 기록

## [09:30] SBR 공정 스크립트 r2 리팩토링

### 한 일 (What)
- `251117_Process_SBR_jyj.py`의 가동시간 계산 로직을 `utils/operation_time.py`로 분리
- 공통 유틸 함수 3개 추출: `calc_aeration_time`, `calc_settling_time`, `calc_decant_time`
- 기존 호출부 12곳 업데이트

### 왜 (Why)
- AMX 공정 스크립트도 같은 로직을 중복 사용 중 → DRY 위반
- 추후 검토자가 로직만 따로 검증할 수 있게 분리 필요

### 의사결정·트레이드오프
- **모듈 분리 vs 인라인 유지:** 분리 선택. 이유: 다른 공정에서 재사용 + 단위 테스트 가능해짐
- **함수명 한글 vs 영문:** 영문. 이유: `01. 영문버전/` 라이브러리와 네이밍 통일

### 다음 (Next)
- AMX 공정 스크립트도 동일 리팩토링 필요
- `utils/operation_time.py`에 단위 테스트 추가

---

## [14:15] 가동시간 단위 테스트 작성

### 한 일 (What)
- `tests/test_operation_time.py` 신규
- 3개 함수에 대해 정상/엣지 케이스 12개 작성, 전부 통과

### 왜 (Why)
- 오전 리팩토링으로 분리한 로직의 회귀 방지

### 의사결정·트레이드오프
- pytest vs unittest: pytest 선택. 이유: 기존 프로젝트에서 이미 사용 중
```
