# 파일 포맷 상세 (work-done 스킬)

## 일일 파일 (YYYY-MM-DD.md)

### Frontmatter 스키마

```yaml
---
date: 2026-05-17              # ISO 형식 (YYYY-MM-DD)
topics: [SBR, python-script]  # 주제 태그 배열. 짧고 검색 가능한 키워드
projects: [WAI-Design]        # 프로젝트명 배열. 작업 폴더 경로에서 추론
sessions: 2                   # 이 날 /work_done이 호출된 횟수
---
```

규칙:
- `date`는 무조건 ISO 형식
- `topics`·`projects`는 항상 배열 (`[]` 사용). 단일 값이어도 배열.
- 태그는 공백 대신 하이픈 (`python-script`, `SBR-process`)
- 중복 제거. 정렬은 추가된 순서를 유지해도 OK.

### 본문 구조

```markdown
# YYYY-MM-DD 업무 기록

## [HH:MM] <세션 제목 — 한 일의 핵심을 한 줄로>

### 한 일 (What)
- 동사로 시작하는 구체적 변경/작업
- 파일명·함수명·수치 등 식별자 포함
- 한 줄당 하나의 행동

### 왜 (Why)
- 작업의 동기. 사용자가 명시한 맥락 우선.
- 해결하려던 문제, 사용자의 요청, 외부 제약 등

### 의사결정·트레이드오프
- "A vs B 중 A 선택. 이유: ..." 형태
- 사용자와 의논해서 정한 선택은 반드시 포함
- 미루기로 한 것도 의사결정이므로 적기

### 다음 (Next)
- 후속 작업, 미완료 항목, 사용자가 별도로 챙기기로 한 것
- 없으면 이 섹션 생략 가능
```

여러 세션이 있을 때는 `## [HH:MM]` 섹션을 `---` 구분선으로 나눠 누적:

```markdown
## [09:30] 첫 세션 제목

### 한 일 (What)
...

---

## [14:15] 두 번째 세션 제목

### 한 일 (What)
...
```

### 풀 예시

```markdown
---
date: 2026-05-17
topics: [SBR, python-script, refactor]
projects: [WAI-Design]
sessions: 2
---

# 2026-05-17 업무 기록

## [09:30] SBR 공정 스크립트 r2 리팩토링

### 한 일 (What)
- `251117_Process_SBR_jyj.py`의 가동시간 계산 로직을 `utils/operation_time.py`로 분리
- 공통 유틸 함수 3개 추출: `calc_aeration_time`, `calc_settling_time`, `calc_decant_time`
- 기존 호출부 12곳 업데이트

### 왜 (Why)
- AMX 공정 스크립트도 같은 로직을 중복 사용 중 → DRY 위반
- 추후 검토자가 로직만 따로 검증할 수 있게 분리 필요

### 의사결정·트레이드오프
- **모듈 분리 vs 인라인 유지:** 분리 선택. 이유: 다른 공정에서 재사용 + 단위 테스트 가능해짐
- **함수명 한글 vs 영문:** 영문. 이유: `01. 영문버전/` 라이브러리와 네이밍 통일

### 다음 (Next)
- AMX 공정 스크립트도 동일 리팩토링 필요
- `utils/operation_time.py`에 단위 테스트 추가

---

## [14:15] 가동시간 단위 테스트 작성

### 한 일 (What)
- `tests/test_operation_time.py` 신규
- 3개 함수에 대해 정상/엣지 케이스 12개 작성, 전부 통과

### 왜 (Why)
- 오전 리팩토링으로 분리한 로직의 회귀 방지

### 의사결정·트레이드오프
- pytest vs unittest: pytest 선택. 이유: 기존 프로젝트에서 이미 사용 중
```

## 태그 사전 (_index/tags.md)

`/work_done`이 정규화 단계에서 읽고 쓴다. 사용자도 직접 편집 가능.

### 구조

```markdown
# 태그 사전

> 이 파일은 `/work_done` 스킬이 자동 유지합니다.
> 직접 편집해도 됩니다 — 다음 호출 시 그대로 사용됩니다.
> 형식: `- <표준명> (<동의어1>, <동의어2>, ...)`
> 동의어가 없으면 괄호 생략 가능.

## <카테고리 1>
- 표준명 (동의어1, 동의어2, ...)
- 표준명만

## <카테고리 2>
- ...
```

### 카테고리 (고정 5개)

- **공정** — 수처리 공정 (SBR, AMX 등)
- **작업유형** — 일의 종류 (refactor, docs 등)
- **프로젝트** — 사용자의 프로젝트 (WAI-Design 등)
- **도구** — 라이브러리·툴 (python, excel 등)
- **기타** — 위 4개에 안 맞는 것

### 한 줄 형식 파싱 규칙

```
- SBR (sbr, SBR-process, 연속회분식)
  └ 표준명: "SBR"
  └ 동의어: ["sbr", "SBR-process", "연속회분식"]

- python
  └ 표준명: "python"
  └ 동의어: [] (괄호 없음)
```

### 매칭 우선순위 (3.5단계에서 사용)

1. 표준명 정확 일치 (대소문자 무시)
2. 동의어 정확 일치 (대소문자 무시)
3. 퍼지 매칭 — 편집 거리 ≤ 2 또는 부분 문자열 포함 → 사용자 확인
4. 매칭 실패 → 신규 태그로 카테고리 묻기

### 자동 갱신

- **신규 표준명:** 해당 카테고리 끝에 `- <표준명>` 추가
- **신규 동의어:** 기존 줄의 괄호 안에 추가 (괄호 없었으면 새로 만듦)
- **카테고리 이동·삭제·이름 변경:** 사용자 수동 편집 (다음 호출 시 그대로 반영)
